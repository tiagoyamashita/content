データベースのボトルネック

**データベース**は、Web システムで最も一般的なボトルネックです (読み取り、書き込み、ロック、**接続プール**)。

## 1. 読み取りのボトルネック

|問題 |信号 |修正 |
|----------|----------|-----|
|フルテーブルスキャン | `EXPLAIN` → シーケンススキャン |述語の索引 |
| **N+1 クエリ** | ORM: 1 + N クエリ | JOIN、バッチ `IN`、データローダー |
|レプリカのラグ |レプリカ上の古い読み取り |クリティカル読み取り → プライマリ。モニターの遅れ |
|長いトランザクションは読み取りをブロックします。ロック待機 |短いTX;コミット済み / MVCC を読む |

## 2. 書き込みのボトルネック

|問題 |修正 |
|----------|-----|
|単一のプライマリ上限 (~10,000 ～ 50,000 書き込み/秒 Postgres) |シャード;非同期キュー; CQRS |
|インデックス書き込み増幅 |インデックスが少なくなります。部分インデックス |
|デッドロック |一貫したロック順序。楽観的ロック (CAS) |
| WAL / ディスク |より高速なストレージ。チェックポイントを調整する |

## 3. 接続プールの枯渇

```text
500 pods × 10 connections = 5 000  →  DB max_connections = 100  →  crash
```

|修正 |役割 |
|-----|------|
| **PgBouncer** / **RDS プロキシ** |多数のアプリ接続 → 少数の DB 接続を多重化 |
|適切なサイズのプール |経験則: ~(2 × CPU コア) + インスタンスごとのディスク スピンドル |
|短いクエリ |すぐに接続を解除してください |

## 4. クエリ最適化チェックリスト

- 遅いクエリの場合は [ ] `EXPLAIN (ANALYZE, BUFFERS)`
- [ ] 複数列 WHERE / ORDER BY の複合インデックス
- [ ] **インデックスのカバー** — インデックスのみのスキャン
- [ ] **部分インデックス** — `WHERE active = true`
- [ ] 重い集約の具体化されたビュー
- [ ] **プルーン + アーカイブ用に日付ごとにパーティション**

## 5. インデックスが追加されましたが、まだ遅いですか?

|チェック | |
|------|---|
|複合インデックス内の列の順序が間違っています | |
|列 (`WHERE LOWER(email)`) の関数 - 式インデックスが必要 | |
|統計が古い — `ANALYZE` | |
|行数の推定値が間違っています - 統計目標を増加します | |
|ディスクへの流出のソート — work_mem / ORDER BY のインデックス | |
|クエリ プランではなくロック待機 | |

## 6. 読み取りと書き込みのスケーリング パス

|パス |いつ |
|------|----------|
|リードレプリカ |読み取りが多い。ラグを許容する |
|キャッシュ (Redis) |ホットキー、繰り返し読み取り |
|シャーディング |書き込みスケールが単一プライマリを超える |
|非正規化 |読み取りパスが安くなります。複雑さを書き出す |
