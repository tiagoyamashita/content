Key-Value ストア

**キー/値** データベースは、不透明な **キー** (文字列、バイト) を **値** (文字列、バイト、または構造化 BLOB) にマップします。通常、ルックアップは **O(1)** またはそれに近い値です。製品がセカンダリ インデックス レイヤーを追加しない限り、任意のフィールド検索用の **クエリ言語**はありません。

## 1. データモデル

```text
key                    value
─────────────────────  ─────────────────────────
session:abc123         { "userId": 42, "cart": [...] }
user:42:profile        "displayName=Ada"
feature:dark_mode      "on"
rate_limit:192.168.1.1 "37"
```

アプリケーションは **キーに名前を付け**、**値を解釈します**。ストアは JSON スキーマを理解しません。

## 2. 操作

|操作 |意味 |
|----------|----------|
| **GET** / **SET** | 1 つのキーの読み取りまたは書き込み |
| **DELETE** |キーを削除 |
| **INCR** |アトミック整数バンプ (カウンター) |
| **EXPIRE / TTL** |キーは時間が経つと消えます - セッションやキャッシュに最適 |
| **CAS** |比較して設定 — 値が変更されていない場合にのみ書き込みます (楽観的ロック) |

Redis の例 (CLI):

```text
SET session:abc123 "{\"userId\":42}" EX 3600
GET session:abc123
INCR rate_limit:10.0.0.1
EXPIRE rate_limit:10.0.0.1 60
```

## 3. インメモリ vs 耐久性

|スタイル |行動 |例 |
|----------|----------|----------|
| **インメモリ プライマリ** |速い;スナップショットまたは耐久性のための AOF | Redis |
| **ディスクバックアップ、分散型** |ノード間のパーティション化されたキー | DynamoDB、Riak |
| **コーディネート** |小さなキー、ロックの強力な一貫性 | etcd、ZooKeeper |

**Redis** は、PostgreSQL の前にある **キャッシュ** であることが多く、障害時の損失を受け入れて設計しない限り、記録システムではありません。

## 4. よくあるパターン

### キャッシュアサイド

```text
1. READ:  app → cache GET key
           miss → DB → SET cache → return
2. WRITE: app → DB commit → DELETE cache key (or update)
```

キャッシュを無効にせずに DB を更新すると、キャッシュが古くなります。コードで順序を定義します。

### セッションストア

Web サーバーはセッション ID を Cookie に保存します。 Redis の **session:{id}** はユーザー状態 (スティッキー セッションなしの水平スケーリング) を保持します。

### レート制限

IP + **EXPIRE** ウィンドウごとの **INCR** キー — 単純な固定ウィンドウ リミッター。

### 分散ロック (注意)

**SET キー値 NX EX ttl** — ホルダーは 1 つだけ。本番環境では有効期限とフェンシングトークンを処理する必要があります。

## 5. Java スケッチ (レタス / Redis)

```java
// Compile: javac --release 22 …
// Dependency: io.lettuce:lettuce-core (conceptual)
try (var redis = RedisClient.create("redis://localhost:6379").connect()) {
  var commands = redis.sync();
  commands.setex("session:abc", 3600, "{\"userId\":42}");
  String json = commands.get("session:abc");
}
```

Redis を使用した Spring Cache は **`@Cacheable`** メソッドをサポートします。これはフレームワーク層での同じキーと値の考え方です。

## 6. 強みと限界

**強み**

- 極めて低いレイテンシ**の読み取り/書き込み
- **シンプルなメンタル モデル** — 1 つのキー、1 つの BLOB
- **TTL** 内蔵
- **アトミック** カウンターとリスト (Redis データ構造: ハッシュ、セット、ソート セット)

**制限**

- いいえ **JOIN** - アクセス パスに一致するキーを設計します
- 純粋な RAM ストアの **メモリ コスト**
- **ホットキー** — 1 つの人気のあるキーがシャードを飽和させます
- **重複のないリレーショナル レポートの代替品ではありません**

## 7. Key-Value を選択する場合

- **キャッシュ** レイヤー (HTML フラグメント、クエリ結果、オブジェクト)
- **セッション**、**レート制限**、**機能フラグ**
- **リーダーボード** (Redis のソートされたセット)
- **パブ/サブ** および単純なキュー (耐久性には十分注意してください)
