アプリケーションとネットワークセキュリティ
**アプリケーションセキュリティ**はバグが悪用に変わるのを防ぎます。**ネットワークセキュリティ**は誰が何に到達できるかを制限します。併せて、障害時の**爆発半径**を小さくします。

## 1. OWASP Top 10（考え方）

[OWASP Top 10](https://owasp.org/www-project-top-ten/) は時代とともに変わりますが、**テーマ**は一定です。

| テーマ | 何が起きるか | 開発の習慣 |
|-------|-----------------|-----------|
| **アクセス制御の不備** | ユーザー A が B の注文を読む（IDOR） | すべてのオブジェクトアクセスで所有者を確認 |
| **インジェクション** | ユーザー入力から SQL/シェルを組み立て | パラメータ化クエリ。文字列連結しない |
| **暗号の失敗** | 弱い TLS、ログ内のパスワード | 現代的暗号。シークレットをログに出さない |
| **不安全な設計** | ログインにレート制限なし | 設計時に脅威モデル |
| **設定ミス** | デフォルト admin パスワード、公開 S3 | ハードニングガイド、IaC レビュー |
| **脆弱なコンポーネント** | Log4Shell 型 CVE | ロックファイル、Dependabot、CI スキャン |
| **認証の失敗** | 弱いセッション、admin に MFA なし | 中央 IdP、安全な Cookie |
| **完全性の失敗** | 未署名更新、改ざん CI | アーティファクト署名、デプロイ検証 |
| **ログの欠如** | 不正後の監査証跡なし | 構造化セキュリティログ |
| **SSRF** | サーバーが内部 URL を取得 | アウトバウンド許可リスト。生のユーザー URL を使わない |

多くの行をカバーする**セキュアコーディング**のルール: **クライアント入力を信頼しない** — サーバー側で検証・エンコード・認可する。

## 2. 入力検証と出力エンコード

| 層 | ルール |
|-------|------|
| **API** | スキーマ検証（型、長さ、列挙） |
| **SQL** | プリペアドステートメント / ORM のパラメータバインド |
| **HTML** | コンテキストに応じたエンコード（XSS 回避） |
| **Shell** | ユーザー文字列を `exec` に渡さない |
| **ファイルアップロード** | 型・サイズ制限。Web ルート外に保存 |

```text
Client sends:  { "role": "admin" }  on signup
Server must:  ignore role field OR reject — don't trust hidden form fields
```

## 3. TLS と転送層

| 実践 | 詳細 |
|----------|--------|
| **HTTPS 全体** | HTTP → HTTPS リダイレクト。公開サイトは HSTS |
| **TLS バージョン** | 最低 1.2。1.3 を推奨 |
| **証明書管理** | Let's Encrypt / ACM / cert-manager — 更新自動化 |
| **mTLS** | サービス間: 双方が証明書を提示 |

TLS は**通信路上**を守ります。アプリの認証バグや侵害済みサーバーは**直しません**。

## 4. ネットワークセグメンテーション

```text
Internet
  → DMZ (load balancer, WAF)
  → App tier (no direct DB port to internet)
  → Data tier (private subnet only)
  → Admin / bastion (jump host, VPN, or SSO portal)
```

| コントロール | 目的 |
|---------|---------|
| **セキュリティグループ / NSG** | 層間で必要なポートのみ許可 |
| **プライベートサブネット** | DB と内部 API は公開ネットから到達不可 |
| **エグレスフィルタ** | 侵害されたアプリの外向き通信を制限 |
| **WAF** | エッジで一般的な HTTP 攻撃をブロック |

ネットワークの詳細: [CS101 ネットワーク](../CS101/networking/i-tcp-udp-and-transport-basics.md)。

## 5. ゼロトラスト（実践版）

**「信頼せず、常に検証」** — 社内ネットワークにいるだけでは**十分ではない**。

| 旧モデル | ゼロトラスト |
|-----------|------------|
| VPN = 内側 = 信頼 | すべてのリクエストを認証 + 認可 |
| フラットな内部ネット | マイクロセグメンテーション |
| 長寿命 VPN セッション | 短命のデバイス + ユーザーコンテキスト |

| ステップ | アクション |
|------|--------|
| 1 | 強いアイデンティティ（SSO + MFA） |
| 2 | 可能ならデバイス状態（管理端末、パッチレベル） |
| 3 | アプリ/リソースごとのポリシー（「全社員が本番 DB を見る」ではない） |
| 4 | 異常のログとアラート |

初日からベンダーの「ゼロトラストプラットフォーム」は必須ではありません。**リクエストごとの ID + 最小ネットワークアクセス**が核心のシフトです。

## 6. API と B2B セキュリティ

| リスク | 緩和 |
|------|------------|
| オブジェクトレベル認可の不備 | リソース ID をテナント/ユーザーにスコープ |
| 過剰なデータ露出 | DTO — 内部モデル全体を返さない |
| マスアサインメント | 更新可能フィールドの許可リスト |
| レート制限 | IP、API キー、ユーザーごと |
| Webhook | HMAC 署名検証。リプレイ防止 |

公開 API: バージョン、認証の文書化、悪用パターンの監視。

## 7. 依存関係とビルドセキュリティ

| 段階 | コントロール |
|-------|---------|
| **ソース** | ブランチ保護、必須レビュー |
| **依存関係** | ロックファイル、CI の CVE ゲート |
| **ビルド** | 再現可能ビルド、固定ベースイメージ |
| **デプロイ** | 署名コンテナ、本番前ポリシー |

パイプラインの詳細: [サプライチェーンと SLSA](../sre101/cicd/security-and-best-practices/ii-supply-chain-and-slsa.md)。

## 8. セキュリティヘッダー（Web アプリ）

| ヘッダー | 効果 |
|--------|--------|
| `Content-Security-Policy` | スクリプトソース制限 — XSS 緩和 |
| `X-Frame-Options` / `frame-ancestors` | クリックジャッキング防止 |
| `Strict-Transport-Security` | HTTPS 強制 |
| `Referrer-Policy` | Referer に漏れる URL を制限 |

フレームワークミドルウェアまたはリバースプロキシのデフォルトを使い、正当なスクリプトを壊さないよう CSP を調整します。
