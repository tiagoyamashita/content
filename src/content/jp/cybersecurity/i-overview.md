Cybersecurity — 概要
**サイバーセキュリティ（Cybersecurity）** は、**システム・データ・人**を不正アクセス・悪用・損失から守りながら、正当なユーザーにサービスを**利用可能**に保つことです。

このトラックは実践向けです。脅威の考え方、認証・認可、アプリとネットワークの防御、インシデント発生時の対応を扱います。

## 1. CIA トライアド

| 柱 | 意味 | 失敗例 |
|--------|---------|-----------------|
| **機密性（Confidentiality）** | 許可された主体だけがデータを見られる | 顧客 DB の漏洩 |
| **完全性（Integrity）** | データとシステムが改ざんされない | 管理画面で価格を書き換え |
| **可用性（Availability）** | 必要なときサービスが動く | DDoS でチェックアウト停止 |

現代のセキュリティでは次も加わります。

| 拡張 | 理由 |
|-----------|-----|
| **認証（Authentication）** | 呼び出し元が**誰か**を証明 |
| **認可（Authorization）** | **何をしてよいか**を決定 |
| **否認防止（Non-repudiation）** | 誰がいつ何をしたかの監査証跡 |
| **プライバシー** | データの収集・利用の適法性（GDPR など） |

## 2. 多層防御（Defense in depth）

単一の対策ではすべての攻撃を止められません。一層が破られても全体が陥落しないよう、層を重ねます。

```text
User → WAF / CDN → TLS → auth → app validation → DB permissions → encryption at rest → backups → monitoring
```

| 層 | 例 |
|-------|----------|
| **境界** | ファイアウォール、DDoS 対策、地域ブロック |
| **アイデンティティ** | SSO、MFA、スコープ付きサービスアカウント |
| **アプリケーション** | 入力検証、パラメータ化クエリ、CSRF トークン |
| **データ** | 暗号化、トークン化、行レベルセキュリティ |
| **運用** | パッチ、ログ、アラート、インシデント手順書 |

## 3. 誰から守るか

| 主体 | 動機 | 典型手法 |
|-------|------------|-------------------|
| **機会主義的攻撃者** | 楽な金（ランサム、カード詐欺） | フィッシング、公開 S3 |
| **インサイダー** | 復讐、利益、不注意 | 過剰権限、データ持ち出し |
| **組織犯罪** | 大規模詐欺 | クレデンシャルスタッフィング、マルウェア |
| **国家支援** | スパイ、破壊 | サプライチェーン、ゼロデイ |
| **スクリプトキッディ** | 好奇心 | デフォルトパスワードのスキャン |

すべてのシステムで全主体に同等の防御は不可能です。**リスクの優先付け**（次のノート）で努力の配分を決めます。

## 4. セキュリティと利便性

| 緩すぎ | 厳しすぎ |
|-----------|-----------|
| 共有 admin パスワード | 社内 Wiki の毎クリックに MFA |
| ログなし | dev の読み取りに 6 か月の承認 |

**比例した**対策を目指します。最重要資産を最も厳しく守り、TLS 全体適用や CI のシークレットスキャンなど安全なデフォルトを自動化します。

## 5. ソフトウェアキャリアでの位置づけ

```text
機能開発     →  セキュアコーディング（検証、シークレット、依存関係）
本番リリース   →  IAM、ネットワークポリシー、CI/CD 署名（SRE トラック）
本番運用     →  監視、IR、パッチ（このトラック + SRE）
```

開発者は**脆弱性になるバグ**を担います。SRE とセキュリティチームは**プラットフォームのガードレール**を担いますが、**インシデント対応**は全員の共有です。
